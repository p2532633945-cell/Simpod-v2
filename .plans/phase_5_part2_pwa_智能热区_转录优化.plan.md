### P6-1: PWA 手机端支持（2-3 day）

**目标**：用户可以把 Simpod 安装到手机主屏幕，像 App 一样使用

**功能需求**：
1. `manifest.json` — 应用名称、图标、主题色、启动画面
2. Service Worker — 离线缓存静态资源（页面、JS、CSS）
3. 手机端播放器布局优化

4. iOS Safari 特殊处理
   - `apple-touch-icon`
   - `apple-mobile-web-app-capable`

**技术方案**：next-pwa 或手动配置 next.js manifest

**验收标准**：
- [ ] Chrome/Safari 显示「添加到主屏幕」提示

- [ ] 断网时静态页面仍可加载

### P6-2: 中途回溯开关（1-2 day）

**目标**：泛听过程中可开启「热区即时回溯」模式，每次生成热区后自动跳回该热区起点重听

**背景**：
- 默认模式：泛听 → 批量复盘（已有）
- 新增模式：泛听 → 遇到难点立刻 MARK → 音频自动回到热区起点 → 再听一遍
- 适合场景：有完整转录文本的节目（省去转录等待）

**功能需求**：
1. 播放器界面添加「即时回溯」开关（toggle）
2. 开启时，MARK 生成热区后：
   - 热区生成完成 → 自动 seek 到 `hotzone.start_time`
   - 继续播放（不暂停）
   - 侧边栏高亮刚生成的热区
3. 开启条件提示：
   - 若节目有预加载转录文本 → 直接可用（无等待）
   - 若无转录 → 提示「转录生成中，生成完成后自动跳回」
4. 心流保护：
   - 热区生成 < 2s 时静默回溯
   - 热区生成 > 2s 时先继续播放，生成完成后下一次 MARK 时再提示

**技术实现**：
- `playerStore` 添加 `instantReplayMode: boolean`
- `PodcastPlayerPage` 的 `processAnchorsToHotzones` 回调中，若模式开启则在 resolve 后调用 `onSeek(hotzone.start_time)`

**验收标准**：
- [ ] 开关显示在播放器界面，状态持久化（localStorage）
- [ ] 开启后 MARK → 自动回溯到热区起点
- [ ] 关闭后恢复正常泛听模式
- [ ] 热区生成过慢时不打断播放

### P6-3: 智能热区时间范围（档位调节）（1.5 day）

**目标**：用户可以调节热区「语义边界扩展档位」，控制热区上下文范围

**POC 说明**（先验证可行性再完整实现）：
> POC = Proof of Concept，即「概念验证」。先用最小代价验证技术方案是否可行，再决定是否完整开发。

**档位设计**：

| 档位 | 名称 | 扩展逻辑 | 适用场景 |
|------|------|---------|----------|
| 🔴 Tight | 精准 | 仅保留触发词所在句子（±3s） | 单个生僻词、专业术语 |
| 🟡 Normal | 标准 | 当前句子 + 前后各一句（±10s，现有默认） | 一般难点 |
| 🟢 Wide | 宽松 | 整个语义段落（±20s，基于标点/停顿检测） | 需要上下文铺垫的难点 |

**技术实现**：
1. 在 `playerStore` 添加 `hotzoneRange: 'tight' | 'normal' | 'wide'`
2. 在播放器界面添加档位选择器（3个按钮）
3. 修改 `hotzone.ts` 的 `processAnchorsToHotzones`：
   - tight: buffer = 3s
   - normal: buffer = 10s（现有）
   - wide: buffer = 20s
4. POC 阶段：先只实现 buffer 秒数调节，不做 NLP 语义分析
5. 后续（Phase 6）：接入 NLP 做真正的语义边界检测

**验收标准**：
- [ ] 档位选择器显示在播放器 MARK 按钮附近
- [ ] 切换档位后新建热区使用对应 buffer
- [ ] 旧热区不受影响

### P6-4: 转录来源优化（1.5 day）

**目标**：减少 Groq API 调用，提升转录质量和响应速度

**背景分析（你提到的三个来源）**：

#### 来源 1：通用转录缓存（已有，需确认）
当前架构中 `transcripts` 表已经存在：
```
audio_id + start_time + end_time → text + words
```
✅ 已实现：同一音频片段只转录一次，多用户共享缓存
❓ 待确认：缓存命中率是否正常工作

#### 来源 2：RSS Feed 官方转录文本
部分播客在 RSS feed 中包含官方字幕/转录（`<podcast:transcript>` 标签，Podcast Namespace 标准）：
- 格式：SRT、VTT、JSON
- 质量：最高（官方人工校对）
- 代表性播客：NPR、BBC、Spotify 原创

**实现计划**：
- 在 `rss-parser-v2.ts` 解析 `<podcast:transcript>` 标签
- 若存在，下载并解析转录文件
- 存入 `transcripts` 表，标记 `source: 'official'`
- 播放时优先使用 official 转录，跳过 Groq API

#### 来源 3：第三方高质量转录数据库
热门英语播客转录来源：
- **Podscribe**（podscribe.com）— 商业 API，有 1M+ 剧集转录
- **Listen Notes**（listennotes.com）— 有部分转录数据
- **Whisper 社区数据集**（OpenAI Whisper 大规模转录项目）
- **直接爬取**：部分播客官网提供完整文字稿（如 Lex Fridman、Tim Ferriss）

**实现计划（分阶段）**：
1. **Phase 5 Part 2**（现在做）：
   - 实现 RSS 官方转录解析（低成本，高价值）
   - 添加转录来源标记（`official` / `groq` / `cached`）
   - 在 UI 显示转录来源（让用户知道质量）

2. **Phase 6**（后续做）：
   - 接入 Podscribe API 查询现有转录
   - 实现「转录预加载」：进入播放页面时后台预取全集转录
   - 建立内部转录数据库（热门播客优先）

**验收标准**：
- [ ] RSS 解析时检测并存储官方转录
- [ ] 热区生成时优先使用已有转录（命中率提升）
- [ ] 转录来源在 UI 中可见
