# Phase 6：核心体验极致化 — 详细执行计划

> **目标**：转录质量 + 泛听体验 + 回溯精准 + 定位智能
> **周期**：3-4 周
> **成功指标**：转录准确率 > 95%，回溯精准度 > 90%，定位响应 < 200ms

---

## 🎯 四大核心维度

### 维度 1：转录质量优化

**目标**：从"能用"→ "可信"

#### 1.1 官方转录识别与优先级
```
优先级：官方转录 > Groq 转录 > 用户编辑
```

**任务**：
- [ ] 在 RSS 解析时标记转录来源（official / groq / user）
- [ ] 播放器 UI 显示转录来源图标（✓ 官方 / ⚡ Groq / ✏️ 用户）
- [ ] 数据库 hotzones 表添加 `transcript_source` 字段
- [ ] 复盘页面按来源过滤显示

**预期收益**：用户知道转录质量来源，建立信任

---

#### 1.2 多模型转录对比
```
当前：仅用 Groq Whisper
目标：Groq + 备选模型对比
```

**任务**：
- [ ] 集成 OpenAI Whisper API（备选）
- [ ] 设计对比逻辑：
  - 同一音频片段用两个模型转录
  - 计算相似度（Levenshtein distance）
  - 相似度 > 90% → 高置信度
  - 相似度 < 70% → 标记为"需人工审核"
- [ ] 在 hotzones 表添加 `transcript_confidence` 字段（0-100）
- [ ] UI 显示置信度指示（绿/黄/红）

**预期收益**：自动质量评分，用户可信度 ↑↑

---

#### 1.3 转录缓存策略优化
```
当前：按 audio_id 缓存
目标：按内容哈希 + 时间戳缓存
```

**任务**：
- [ ] 计算音频片段的内容哈希（MD5）
- [ ] 缓存键改为 `hash_timestamp`
- [ ] 设置缓存过期时间（7 天）
- [ ] 实现缓存预热（热门播客预先转录）

**预期收益**：减少 API 调用 30%，成本 ↓

---

#### 1.4 转录质量数据收集
```
收集用户反馈：这段转录准确吗？
```

**任务**：
- [ ] 在每个热区卡片添加"转录质量反馈"按钮（👍 准确 / 👎 有误）
- [ ] 记录反馈到 `transcript_feedback` 表
- [ ] 后台定期分析：哪些模型/播客的转录质量最差
- [ ] 生成质量报告（周报）

**预期收益**：数据驱动的质量改进

---

### 维度 2：泛听体验优化

**目标**：从"能播"→ "舒适"

#### 2.1 后台播放稳定性
```
问题：切换 tab 或锁屏时音频中断
目标：后台播放不中断
```

**任务**：
- [ ] 检查 `visibilitychange` 事件处理（不应该 pause）
- [ ] 添加 `pagehide` / `pageshow` 事件监听
- [ ] 测试：锁屏 → 解锁，音频继续播放
- [ ] 测试：切换 tab → 返回，音频继续播放
- [ ] iOS Safari 特殊处理（需要 `audio` 元素在 DOM 中）

**预期收益**：用户可以后台听播客

---

#### 2.2 倍速播放优化
```
当前：0.5x - 2x，6 档
目标：更细粒度 + 记忆用户偏好
```

**任务**：
- [ ] 添加 0.8x, 1.1x, 1.3x 等中间档位
- [ ] localStorage 记录用户最常用倍速
- [ ] 每次进入播放器自动应用上次倍速
- [ ] 倍速按钮显示当前倍速（更明显）

**预期收益**：用户体验个性化

---

#### 2.3 音量和音质优化
```
问题：某些播客音量太小/太大
目标：自动音量归一化
```

**任务**：
- [ ] 实现音量归一化（loudness normalization）
  - 使用 Web Audio API 的 DynamicsCompressor
  - 目标 LUFS: -14（播客标准）
- [ ] 添加"音量归一化"开关（默认开启）
- [ ] 测试：不同播客音量是否一致

**预期收益**：听觉舒适度 ↑

---

#### 2.4 播放流畅性
```
问题：某些 URL 加载慢，音频卡顿
目标：预加载 + 缓冲优化
```

**任务**：
- [ ] 实现音频预加载（preload="metadata" → "auto"）
- [ ] 添加缓冲进度显示
- [ ] 实现自适应比特率（如果支持）
- [ ] 测试：弱网环境下的播放体验

**预期收益**：播放稳定性 ↑

---

### 维度 3：回溯精准优化

**目标**：从"能跳"→ "精准"

#### 3.1 智能时间范围（上下文感知）
```
当前：固定 ±3s/10s/20s
目标：根据内容自动调整
```

**任务**：
- [ ] 分析转录文本，识别句子边界
- [ ] 当用户 MARK 时，自动扩展到完整句子
- [ ] 例：用户在"world"处 MARK
  - 原始：±10s
  - 智能：扩展到完整句子 "The world is changing" (可能 ±15s)
- [ ] 添加"自动调整"开关（默认开启）

**任务分解**：
- [ ] 实现句子边界检测（使用 NLP 库或正则）
- [ ] 在 `processAnchorsToHotzones` 中集成
- [ ] 测试：不同语言/口音的句子识别准确率

**预期收益**：热区更自然，减少用户手动调整

---

#### 3.2 快速定位优化
```
当前：按钮 ◁15s / 15s▷
目标：快速定位 + 智能推荐
```

**任务**：
- [ ] 添加"快速定位"快捷键：
  - `J` / `K`：上一个/下一个热区
  - `U` / `I`：上一个/下一个 Pending 热区
  - `O` / `P`：上一个/下一个 Reviewed 热区
- [ ] 显示快捷键提示（首次使用时）
- [ ] 测试：快捷键响应时间 < 100ms

**预期收益**：键盘党用户体验 ↑↑

---

#### 3.3 回溯历史记录
```
新功能：记录用户的回溯操作
```

**任务**：
- [ ] 添加"回溯历史"面板
- [ ] 记录最近 20 次跳转操作（时间戳 + 目标时间）
- [ ] 显示"最常回溯的时间点"
- [ ] 一键返回上一个位置（Undo）

**预期收益**：用户可以快速回到之前的位置

---

### 维度 4：定位智能优化

**目标**：从"能搜"→ "精准"

#### 4.1 模糊定位（Fuzzy Search）
```
当前：精确匹配转录文本
目标：模糊匹配 + 拼写容错
```

**任务**：
- [ ] 实现 Levenshtein distance 模糊匹配
- [ ] 支持拼写错误容错（例：搜"wrld"能找到"world"）
- [ ] 搜索结果按相似度排序
- [ ] 测试：常见拼写错误的匹配准确率

**任务分解**：
- [ ] 在 `TranscriptSearch` 组件中集成模糊匹配
- [ ] 使用 `fuse.js` 库（轻量级）
- [ ] 性能测试：搜索响应时间 < 200ms

**预期收益**：用户不需要精确拼写

---

#### 4.2 语义搜索（Semantic Search）
```
新功能：理解搜索意图，而不仅仅是关键词
```

**任务**：
- [ ] 集成向量化模型（如 Sentence Transformers）
- [ ] 将转录文本转换为向量
- [ ] 用户搜索时也转换为向量
- [ ] 计算余弦相似度，返回语义相关的热区
- [ ] 例：搜"学习方法" → 找到"如何高效学习"的热区

**任务分解**：
- [ ] 选择模型：使用开源 `all-MiniLM-L6-v2`（轻量）
- [ ] 后端实现向量化 API
- [ ] 前端集成搜索
- [ ] 性能测试：向量计算时间 < 500ms

**预期收益**：搜索结果更相关，用户满意度 ↑↑

---

#### 4.3 上下文推荐
```
新功能：基于当前热区推荐相关热区
```

**任务**：
- [ ] 当用户查看一个热区时，显示"相关热区"
- [ ] 使用语义相似度计算相关性
- [ ] 显示 Top 3 相关热区
- [ ] 一键跳转

**预期收益**：帮助用户发现相关内容

---

#### 4.4 搜索历史 + 智能建议
```
当前：简单的搜索历史
目标：智能建议 + 热门搜索
```

**任务**：
- [ ] 记录用户搜索历史（localStorage）
- [ ] 显示"热门搜索"（全用户）
- [ ] 显示"你的搜索"（个人历史）
- [ ] 搜索框输入时自动补全建议

**预期收益**：降低搜索成本
