# 规划新任务

## 功能：$ARGUMENTS

## 使命

将功能请求转化为**全面的实施计划**，通过系统的代码库分析、外部研究和战略规划。

**核心原则**：我们不在此阶段编写代码。我们的目标是创建一个上下文丰富的实施计划，使 AI agents 能够一次性实施成功。

**关键理念**：上下文为王。计划必须包含实施所需的所有信息——模式、必读文档、文档、验证命令——以便执行代理在首次尝试时成功。

**重要：与 PROGRESS.md 协调**

## 第一步：检查 PROGRESS.md

在开始规划前，**必须先读取 PROGRESS.md**：

```bash
读取 PROGRESS.md
```

### 协调规则

1. **如果 $ARGUMENTS 对应 PROGRESS.md 中的待办事项**：
   - 在计划中明确引用该任务
   - 说明计划将实现该待办事项
   - 保持与 PROGRESS.md 中的描述一致

2. **如果 $ARGUMENTS 是新任务（不在 PROGRESS.md 中）**：
   - 检查是否应该先添加到 PROGRESS.md
   - 如果是高优先级任务，先建议用户更新 PROGRESS.md
   - 在计划中说明这是临时规划任务

### 计划中的 PROGRESS 引用格式

在生成的计划开头添加：

```markdown
## PROGRESS.md 关联

**实现任务**：[从 PROGRESS.md 中复制的任务名称]
**优先级**：[🔥 高 / ⚡ 中 / 📋 低]
**预计完成**：[估计时间]
```

## 规划过程

### 阶段 1：功能理解

**深度功能分析：**

- 提取要解决的核心问题
- 识别用户价值和业务影响
- 确定功能类型：新能力/增强/重构/缺陷修复
- 评估复杂度：低/中/高
- 映射受影响的系统和组件

**创建用户故事格式或完善用户提供的描述：**

```
作为一个 <用户类型>
我想要 <操作/目标>
以便 <收益/价值>
```

### 阶段 2：代码库智能收集

**使用专门的 agents 和并行分析：**

**1. 项目结构分析**

- 检测主要语言、框架和运行时版本
- 映射目录结构和架构模式
- 识别服务/组件边界和集成点
- 定位配置文件（pyproject.toml、package.json 等）
- 查找环境设置和构建过程

**2. 模式识别**（有益时使用专门的子 agents）

- 在代码库中搜索类似实现
- 识别编码约定：
  - 命名模式（CamelCase、snake_case、kebab-case）
  - 文件组织和模块结构
  - 错误处理方法
  - 日志模式和标准
- 提取功能域的通用模式
- 记录要避免的反模式
- 检查 CLAUDE.md 中的项目特定规则和约定

**3. 依赖分析**

- 编录与功能相关的外部库
- 了解库的集成方式（检查导入、配置）
- 在 docs/、ai_docs/、.agents/reference 或 ai-wiki（如可用）中查找相关文档
- 注意库版本和兼容性要求

**4. 测试模式**

- 识别测试框架和结构（pytest、jest 等）
- 查找类似测试示例以供参考
- 了解测试组织（单元测试 vs 集成测试）
- 注意覆盖需求和测试标准

**5. 集成点**

- 识别需要更新的现有文件
- 确定需要创建的新文件及其位置
- 映射路由/API 注册模式
- 了解数据库/模型模式（如适用）
- 识别认证/授权模式（如相关）

**澄清歧义：**

- 如果此时需求不明确，在继续之前请用户澄清
- 获取具体的实施偏好（库、方法、模式）
- 在进行之前解决架构决策

### 阶段 3：外部研究与文档

**有益时使用专门的子 agents 进行外部研究：**

**文档收集：**

- 研究最新库版本和最佳实践
- 查找官方文档及特定章节锚点
- 定位实施示例和教程
- 识别常见陷阱和已知问题
- 检查重大更改和迁移指南

**技术趋势：**

- 研究技术栈的当前最佳实践
- 查找相关博客文章、指南或案例研究
- 识别性能优化模式
- 记录安全考虑事项

**编译研究引用：**

```markdown
## 相关文档

- [库官方文档](https://example.com/docs#section)
  - 特定功能实施指南
  - 原因：需要 X 功能
- [框架指南](https://example.com/guide#integration)
  - 集成模式章节
  - 原因：显示如何连接组件
```

### 阶段 4：深度战略思考

**深入思考：**

- 此功能如何融入现有架构？
- 关键依赖项和操作顺序是什么？
- 可能出什么问题？（边缘情况、竞态条件、错误）
- 如何全面测试？
- 有什么性能影响？
- 有安全考虑吗？
- 此方法的可维护性如何？

**设计决策：**

- 通过清晰理由选择替代方法
- 设计可扩展性和未来修改
- 如需要，规划向后兼容性
- 考虑可扩展性影响

## 成功指标

**一次性实施**：执行代理无需额外研究或澄清即可完成功能

**验证完成**：每个任务至少有一个有效的验证命令

**上下文丰富**：计划通过"无先验知识测试" - 不熟悉代码库的人可以仅使用计划内容实施

**与 PROGRESS 协调**：计划明确说明实现哪个 PROGRESS 任务，并有更新指令
