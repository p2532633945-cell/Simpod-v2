# 创建 PRD：生成产品需求文档

## 概述

基于当前对话上下文和已讨论的需求，生成一份全面的产品需求文档（PRD）。使用下面定义的结构和章节来创建一份专业、详尽的 PRD。

## 输出文件

将 PRD 写入：`$ARGUMENTS`（默认：`PRD.md`）

## PRD 结构

创建结构良好的 PRD，包含以下章节。根据可用信息调整深度和细节：

### 必需章节

**1. 执行摘要**
- 简洁的产品概述（2-3段）
- 核心价值主张
- MVP 目标声明

**2. 使命**
- 产品使命声明
- 核心原则（3-5项关键原则）

**3. 目标用户**
- 主要用户画像
- 技术舒适度水平
- 关键用户需求和痛点

**4. MVP 范围**
- **范围内**：MVP 的核心功能（使用 ✅ 复选框）
- **范围外**：推迟到未来阶段的功能（使用 ❌ 复选框）
- 按类别分组（核心功能、技术、集成、部署）

**5. 用户故事**
- 主要用户故事（5-8个），格式："作为一个[用户类型]，我想要[操作/目标]，以便[收益/价值]"
- 为每个故事包含具体示例
- 如相关，添加技术用户故事

**6. 核心架构与模式**
- 高层架构方法
- 目录结构（如适用）
- 关键设计模式和原则
- 特定技术模式

**7. 工具/功能**
- 详细的功能规格说明
- 如果构建 agent：工具设计，包含目的、操作和关键特性
- 如果构建应用：核心功能分解

**8. 技术栈**
- 后端/前端技术及版本
- 依赖项和库
- 可选依赖项
- 第三方集成

**9. 安全与配置**
- 认证/授权方法
- 配置管理（环境变量、设置）
- 安全范围（范围内和范围外）
- 部署考虑事项

**10. API 规范**（如适用）
- 端点定义
- 请求/响应格式
- 认证需求
- 示例负载

**11. 成功标准**
- MVP 成功定义
- 功能需求（使用 ✅ 复选框）
- 质量指标
- 用户体验目标

**12. 实施阶段**
- 分解为 3-4 个阶段
- 每个阶段包含：目标、交付成果（✅ 复选框）、验证标准
- 现实的时间线估计

**13. 未来考虑**
- MVP 后的增强功能
- 集成机会
- 后续阶段的高级功能

**14. 风险与缓解措施**
- 3-5 项关键风险及具体缓解策略

**15. 附录**（如适用）
- 相关文档
- 关键依赖项及链接
- 仓库/项目结构

## 指令

### 1. 提取需求
- 审阅整个对话历史
- 识别显式需求和隐式需求
- 注意技术约束和偏好
- 捕获用户目标和成功标准

### 2. 综合信息
- 将需求组织到适当的章节中
- 在缺失细节的地方填充合理假设
- 保持章节间的一致性
- 确保技术可行性

### 3. 撰写 PRD
- 使用清晰、专业的语言
- 包含具体示例和细节
- 使用 markdown 格式（标题、列表、代码块、复选框）
- 在有帮助的技术章节添加代码片段
- 保持执行摘要简洁但全面

### 4. 质量检查
- ✅ 所有必需章节都存在
- ✅ 用户故事有明确收益
- ✅ MVP 范围现实且定义明确
- ✅ 技术选择有充分理由
- ✅ 实施阶段可执行
- ✅ 成功标准可衡量
- ✅ 术语使用一致

## 风格指南

- **语调**：专业、清晰、行动导向
- **格式**：广泛使用 markdown（标题、列表、代码块）
- **复选框**：使用 ✅ 表示范围内项，❌ 表示范围外项
- **具体性**：偏好具体示例而非抽象描述
- **长度**：全面但易于浏览（通常 30-60 节内容）

## 输出确认

创建 PRD 后：
1. 确认写入文件的路径
2. 提供 PRD 内容的简要摘要
3. 突出显示因信息缺失而做的任何假设
4. 建议下一步操作（例如：审查、完善、规划）

## 注意事项

- 如果关键信息缺失，生成前请提出澄清问题
- 根据可用细节调整章节深度
- 对于高度技术性产品，强调架构和技术栈
- 对于面向用户的产品，强调用户故事和体验
- 此命令包含完整的 PRD 模板结构 - 无需外部参考
